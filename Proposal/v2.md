# Praktische De-Anonymisierung im Tor-Netzwerk: Timing-Leaks, Traffic-Korrelation und Website-Fingerprinting unter Laborbedingungen

## Ausgangslage und Problemstellung

Tor gilt als sicher, aber theoretische Studien zeigen, dass Timing-Analysen und Traffic-Korrelationen Nutzer deanonymisieren können. Praktische Untersuchungen fehlen, besonders unter kontrollierten, lokalen Bedingungen.

## Shadow Simulationen

Mit dem 2011 vorgestellten (jansen2011shadow) Shadow Netzwerksimulator können tor netzwerke lokal und reproduzierbar simuliert werden. Man kann dabei die gewünschte Netzwerkgrösse bspw. 5% angeben. Je grösser das simulierte Netzwerk desto realistischer das Ergebnis. Traffic congestion etc. auch mitsimuliert wird von etlichen clients.

Es gibt ein Paper zur Simulation von WF Attacken im Shadow Simulationstool. Dabei wurde eine Wikipedia Seite geklont und darauf WF Angriffe simuliert (jansen2023data). Der Source Code ist available, aber relativ komplex.

Simulierte WF Attacken sind oft nicht sehr realistische verglichen mit Real World Szenarien. Dennoch kann es spannend sein zu sehen was der Einfluss von circuit padding unter reproduzierbaren Bedingungen für einen Einfluss hat.

**Forschungsfragen:**

- Setup von jansen2023data reproduzieren
- Wie beeinflusst Circuit Padding die WF Erkennungsrate
- Dummy Traffic?

## Ziel der Arbeit und erwartete Resultate

- **Aufbau der Shadow Simulation analog zu jansen2023** — Grösse des Tor Netzwerkes kleiner als bei jansen2023
- **Website-Fingerprinting lokal testen** — Auswahl von 20–50 Webseiten in lokaler Umgebung oder über Tor-Testnetz. ML-Modelle: einfache CNNs oder RNNs, Training auf lokal generierten Packet-Sequences.
- **Simulation und Einfluss von Circuit Padding** - wie veränndert sich die Erkennungsrate etc.
- **Traffic-Analyse-Framework entwickeln** — Capture von Metadaten: Paketgrössen, Richtungen, Timestamps. Keine Entschlüsselung von Inhalten.
- **Simulation von Timing- und Korrelationsangriffen** — Statistical Disclosure Attacks und einfache Traffic-Korrelation. Evaluation unter kontrollierten Bedingungen, mehrere parallele Clients.

- **Evaluation von Gegenmassnahmen** — Traffic-Padding-Tests: Messen von Angriffsresistenz vs. Performance.
- **Ethische Dokumentation** — Kein Zugriff auf externe Nutzer.

## Gewünschte Methoden, Vorgehen

Im Gegensatz zu Jansen soll ein WF Angriff aus Sicht eines ISP's simuliert werden.

| Phase                                   | Beschreibung                                                                                                                                              |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Phase 1: Shadow Setup**               | Tor-Netzwerk mit tornettools aufsetzen, Wikipedia-Mirror als Zielseiten klonen, wget2-Clients konfigurieren                                               |
| **Phase 2: Baseline Datenerhebung**     | Pcap-Captures an Client-Nodes, Feature-Extraktion (Paketgrössen, Richtungen, Timestamps), WF-Classifier (k-FP, DF) auf Baseline trainieren und evaluieren |
| **Phase 3: Circuit Padding aktivieren** | Tors built-in Padding-Mechanismen konfigurieren (verschiedene Stufen), gleiche Datenerhebung wiederholen                                                  |
| **Phase 4: Vergleich & Analyse**        | Erkennungsraten mit/ohne Padding vergleichen, Overhead messen (Bandbreite, Latenz), Trade-off quantifizieren                                              |
| **Phase 5: Dokumentation**              | Ergebnisse, Limitations (Shadow vs. Real World), ethische Reflexion                                                                                       |


**Innovation:**

Der Einfluss von Circuit Padding unter reproduzierbaren lokalen Bedingungen aufzeigen.
